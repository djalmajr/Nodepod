## Recommended sequence

1. **Security + correctness first** — 001, 002, 003 are the "silently wrong" /
   "unsafe to embed" bugs with the widest blast radius. Do these before anything
   cosmetic.
2. **004, 005, 006** — remaining security hardening and the clean correctness
   fixes. 004 and 002 both edit `static/__sw__.js`; land one then rebase the
   other to avoid a merge conflict (no logical dependency).
3. **007, 008, 009** — polyfill/runtime fidelity; small and independently
   verifiable.
4. **010** — trivial cleanup; do anytime.

## Dependency notes

- No hard code dependencies between plans. The only coupling is 002 and 004 both
  touching `static/__sw__.js` — sequence them to avoid conflicts.
- Several plans note that their *ideal* verification needs a service-worker /
  worker-spawn test harness that does not exist yet (findings TEST-01..04). That
  harness is itself a worthwhile plan (not written here); until it lands, those
  plans verify via `type-check` + `grep` assertions + the existing suite. If you
  want, add an `011-test-harness` plan first and make 002/004/006/009 depend on
  it for full regression coverage.

## Findings considered and not (yet) turned into plans

Recorded so they aren't re-audited from scratch. All are real; they were left
out of this batch for scope, risk, or because they're larger design efforts.

- **Perf: full VFS cloned on every spawn/fork** (`process-manager.ts:88-90`,
  `vfs-bridge.ts:30-63`) — highest-value perf win but HIGH risk (worker
  isolation semantics). Needs the test harness first. Design/spike plan
  recommended before implementation.
- **Perf: SharedVFS linear-scan lookups + append-only, never compacts**
  (`shared-vfs.ts`) — M effort, MED risk; do after a harness exists.
- **Perf: module source parsed 2–3× per load; TS stripped via ~15 regex passes**
  (`script-engine.ts:136-228,1651-1663`) — M effort, MED risk (TLA/ESM edge
  cases, see issue #56).
- **Correctness: SyncChannel slot reuse race** (`sync-channel.ts:98-102`) and
  **WASI fs-proxy triple-post race** (`napi-wasm-worker.ts:960-966`) — real,
  MED risk; hard to verify without concurrency tests. Defer to post-harness.
- **Polyfill: `createCipheriv` always throws** (`crypto.ts:530-544`) — L effort
  (Web Crypto AES wrappers); its own plan when a consumer needs it.
- **Polyfill: sync XHR wasm fallback freezes the tab** (`fs.ts:1391-1394`) — M
  effort, MED risk; couples with the install-time precompile path.
- **Shell: `npm ci` ignores lockfile, `npm audit` always clean, no `2>`
  redirection, `${VAR-default}` empty-as-unset** (`shell/**`) — a cluster of
  shell-fidelity fixes worth their own batch.
- **Tests: no coverage for install pipeline / ProcessManager / SW / SDK spawn**
  — the harness plan referenced above; high leverage as a prerequisite.
- **DX: no lint/formatter/pre-commit; no AGENTS.md; CONTRIBUTING omits
  `npm test`** — low-risk tooling adds.

Nothing was rejected as "not a real problem." Everything above is a genuine
finding deferred by scope.
